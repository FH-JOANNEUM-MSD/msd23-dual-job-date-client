Umfang. Dieses Dokument fasst die agile Lieferung der Dual Job Dating Mobile-App über drei Sprints zusammen. Die Ticket-Liste wurde aus YouTrack exportiert (Projektkürzel JOB). Jeder Sprint wird mit einem Zielsatz, den wichtigsten Kennzahlen sowie den ausgelieferten User Stories und Tasks - gruppiert nach Arbeitsbereich - dargestellt.

Programmübersicht

Die Arbeit war als Kanban-orientiertes SCRUM in drei Sprints organisiert. Sprint 1 hat das technische Fundament gelegt: ein Kotlin Multiplatform Projekt, die Paketstruktur, Dependency Injection, Navigation und die CI/CD-Pipeline zum Google Play Store. Sprint 2 hat die beiden zentralen User Flows umgesetzt: Authentifizierung und das Unternehmens-Swipe-/Voting-Erlebnis, beide gestützt auf einen Ktor-basierten Netzwerk-Layer und eine persistente Session. Sprint 3 hat die restliche Produktoberfläche ergänzt (Unternehmens-Detail, Profil, Termine), die UX poliert, Sicherheit gehärtet und die App für den Release auf Android und iOS vorbereitet - abgeschlossen mit der technischen Übergabe an den Auftraggeber.

Alle acht studierendenseitigen User Stories aus dem Anforderungsdokument sind ausgeliefert und auf den main-Branch gemergt. Die Zuordnung der Stories zu Tickets und Code findet sich auf Seite 2.

Prozess

Das Team hat ein einziges YouTrack-Board mit dem Fluss Waiting → To do → In Progress → Done verwendet. Tickets wurden entweder als User Stories ("Als Studierende möchte ich...") oder als technische Tasks (Architektur, CI, Bugfixes) angelegt. Jeder gemergte Pull Request schließt mindestens ein Ticket; der main-Branch war geschützt und jede Änderung lief über einen PR-Review. Eine Story gilt als abgeschlossen, sobald sie auf main gemergt und das zugehörige Ticket auf Done gesetzt wurde. Releases wurden aus einem langlebigen android-production-Branch geschnitten, der eine GitHub Actions Pipeline auslöst, die ein signiertes AAB baut und in das Google Play Closed Testing hochlädt.

Sprint 1 - Fundament

13 von 13 erledigt

Ziel

Das Kotlin Multiplatform Projekt aufsetzen, Paketstruktur und Architektur festlegen, Dependency Injection und Navigation verdrahten und eine vollautomatisierte Release-Pipeline einrichten, die signierte Android-Bundles in den Google Play Store schiebt - alles vor Beginn der eigentlichen Feature-Arbeit.

Wesentliche Ergebnisse

Leeres, aber architektonisch vollständiges App-Skelett, sauberer Schnitt zwischen feature und core, funktionierende Koin DI in commonMain, typsicherer Navigationsgraph mit Bottom Navigation und ein grüner GitHub Actions Workflow, der bei jedem Push auf android-testing baut, signiert und in das Play Store Internal Testing hochlädt.

Sprint 2 - Kern-Features

32 von 32 erledigt

Ziel

Die beiden zentralen User Flows ausliefern: Authentifizierung (Login + persistente Session) und Unternehmens-Swipe + Voting. Den unterstützenden Netzwerk-Layer (Ktor), das geteilte Theme und die Typografie aufbauen und den Swipe-Flow an das echte Backend anschließen.

Wesentliche Ergebnisse

End-to-End-Login über Supabase Auth mit Auto-Login bei späteren App-Starts, persistente Session-Speicherung auf beiden Plattformen, ein MVI-orientiertes LoginViewModel, ein Tinder-artiger Swipe-Screen mit Vote-Übermittlung an das Backend, das vollständige Design-System (Theme + Typografie + App-Hintergrund) sowie ein nativer Bestätigungsdialog, der aus commonMain heraus erreichbar ist.

Sprint 3 - Politur & Release

55 von 55 erledigt

Ziel

Die restliche Produktoberfläche fertigstellen (Unternehmens-Detail, Profil, Termine), die App lokalisieren, Sicherheit härten, UX polieren und sowohl auf Android (Play Store Closed Testing) als auch auf iOS (App Store / TestFlight) veröffentlichen. Der Sprint endet mit der technischen Übergabe an den Auftraggeber.

Wesentliche Ergebnisse

Volle Feature-Parität auf Android erreicht, App über die CI-Pipeline ins Play Store Closed Testing (Alpha) veröffentlicht. iOS-Distribution läuft (Bundle ID + Provisioning fertig, App Store Listing eingereicht, Gerätetests abgeschlossen). Security-Härtung abgeschlossen (JWT-Logs entfernt, R8 Obfuscation, verschlüsselte Session-Speicherung, Token-Refresh). UX-Politur (Skeleton Loader, Pull-to-Refresh, Offline-Hinweis, Landscape-Unterstützung, Accessibility-Labels) gemergt. Technische Übergabedokumentation in dieser Verzeichnisstruktur (docs/).
