Scope. This document describes the architecture of the Dual Job Dating mobile app: its component layering, technology choices, and the data model of the backend persistence layer. Operational topics (CI/CD, distribution, secrets) are covered in the README and the Technical Handover.

Overview

Dual Job Dating is a cross-platform mobile application built with Kotlin Multiplatform and Compose Multiplatform. All application code lives in commonMain; platform-specific code is limited to the Ktor HTTP engine (OkHttp on Android, Darwin on iOS) and one native dialog on Android. The app talks to two external systems: a custom Go backend exposing the domain API and Supabase for authentication, file storage, and the underlying Postgres database.

Layer responsibilities

LayerResponsibility
feature/*/uiComposable screens, ViewModels, immutable State data classes, sealed Event classes. No business logic in screens.
feature/*/dataRepository interfaces (the only thing a ViewModel depends on) and Service implementations that do raw API calls. DTOs are mapped to domain models exclusively in RepositoryImpl.
core/diKoin AppModule.kt binds all interfaces to implementations. Swapping a service requires no changes elsewhere.
core/navigationSingle NavGraph with @Serializable route objects. Navigation side effects are emitted via MutableSharedFlow from ViewModels.
core/networkKtor HttpClient setup, base URL injection, bearer-token authentication, JSON content negotiation, error mapping.
core/sessionSessionManager persists the JWT and refresh token through Multiplatform Settings for auto-login on subsequent launches.
core/uiDualJobDatingTheme, typography, color palette, shared composables (bottom navigation, app background, dialogs).
androidMain / iosMainActivity / ViewController entry points, the platform-specific Ktor engine factory, and one native confirmation dialog on Android.

Data flow per user interaction

A user action is dispatched as a sealed Event to the screen's onEvent() handler, which calls the ViewModel. The ViewModel invokes its Repository, which delegates to a Service that performs the actual Supabase SDK call or Ktor HTTP request. The result is mapped from DTO to domain model inside the repository and used to update the ViewModel's MutableStateFlow. The screen renders the new state via collectAsState(). One-time side effects (navigation, snackbar) are emitted on a separate MutableSharedFlow consumed inside a LaunchedEffect.

Data Model

Persistence lives in the Supabase-hosted Postgres database. The schema below is reconstructed from the OpenAPI specification (docs/tech/backend_api.yaml) and the DTO classes in the Kotlin codebase. Authentication identities are managed by Supabase Auth; users.id is the same UUID Supabase issues for the authenticated principal.

Figure 2 - Entity Relationship Diagram (crow's foot notation).

Entities

EntityPurpose
usersAuthentication identity (Supabase Auth). Distinguishes role (student / company / admin). One users row corresponds to exactly one students or companies profile.
studentsProfile of a participating student. Holds name, study program, and semester. The user_id links to the authenticated identity.
companiesProfile of a participating company. Holds public description, website, logo, and a semicolon-separated list of additional image URLs. active controls whether the company appears in the swipe stack.
preferencesA single vote of one student on one company. preference_type is one of like, dislike, or neutral. Produced by the swipe screen; consumed by the matching algorithm.
eventsA Job Dating event (date, location, activation flag). Owns the time slots for that day.
slotsA single time slot within an event (start / end timestamp). Many slots per event.
meetingsThe output of the matching process. Assigns one student and one company to one slot. Read-only from the app's perspective and displayed on the Appointments screen.

Cardinalities at a glance

  • users 1 — 1 students  (role = student)
  • users 1 — 1 companies  (role = company)
  • students 1 — M preferences M — 1 companies
  • events 1 — M slots
  • slots 1 — M meetings
  • students 1 — M meetings M — 1 companies
